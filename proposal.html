02 — The Journey

From Clinic to Home
And Back

This is how we see your system working — not as a list of features, but as a living experience that connects the clinic to the patient's home and brings the data back to the therapist.

Stage 01
Assessment

The patient visits Visual House in person. The orthoptist evaluates, diagnoses, and registers the patient in the system with their condition — binocular coordination, tracking deficiency, convergence insufficiency. This step stays human. The platform does not diagnose. It receives the diagnosis, and everything that follows is built on it.

Stage 02
Prescription

The therapist opens the platform and builds a personalized therapy program for this patient. Which exercises, what parameters — speed, size, difficulty — what schedule, how many sessions per week, how long each session. The treatment plan is clinical. The delivery mechanism is digital.

Stage 03
Execution

The patient goes home, logs in, and sees today's session. They follow the program. The exercises run on screen. The system records everything — what was completed, what was missed, how long it took, when it happened. For children, the experience must be simple enough to follow with minimal parent help. The system does not require decisions from the patient — just participation.

Stage 04
Insight

The therapist opens their dashboard and the story is there. Who is this patient, what was the diagnosis, what program are they on, what is the adherence, what is the progress. The therapist reviews, adjusts parameters, progresses or pauses the program, or flags for an in-person reassessment. The clinical judgment stays with the professional. The platform gives them the information to make that judgment confidently.

This cycle repeats. Every session generates data. Every data point informs the next clinical decision. The system does not replace the therapist — it extends their reach beyond the walls of the clinic.

03 — Key Observations

What Others
Will Miss

Five things we found in your brief that most developers will overlook entirely.
01
Critical FindingThe Exercise Engine Is The Product
Most proposals will treat the interactive exercise engine as a feature. It is not. It is the core of everything. This engine must render smooth, precise visual stimuli in real time — with exact timing, configurable parameters, and zero lag — because children are using it for therapy. A stuttering animation is not a bug. It is a failed treatment session. We know exactly what this requires technically and we address it in the architecture section below. Getting this wrong means the therapy does not work. Getting it right means you have something licensable to other providers worldwide.
02
UX ArchitectureTwo Completely Different Users In One System
70% of your patients are children. They need an interface so simple and engaging they can run a session with minimal supervision. Your orthoptists need a panel sophisticated enough for clinical decision-making. These are not two pages — they are two entirely different UX philosophies that must coexist in one system. Most developers will build one and compromise the other. And there is an invisible user the brief does not name — the parent. A child doing exercises at home means someone nearby may need to help start a session, understand what is expected, or troubleshoot. The interface must account for this without adding complexity.
03
Compliance RiskSensitive Health Data From Day One
Basic medical history means data protection compliance — including Ley 1581 de 2012 (Colombian personal data protection) — is not a Phase 2 concern. Role-based authentication, encryption, audit trails, and daily backups must be architected into the foundation from day one, not bolted on later at higher cost. Compliance is not a phase. It is a foundation.
04
Strategic Intent"Web PC Only, Chrome" Is Clinically Intentional
This is not a technical limitation. This is a deliberate constraint that protects the precision of your exercise rendering. Visual therapy stimuli require consistent, predictable screen behavior. We will build for this constraint deliberately — not treat it as something to work around in a future phase.
05
Future ArchitectureThe Licensing Ambition Changes How We Build Now
You mentioned future licensing to other healthcare providers and expansion to other countries. This has architectural implications now — even at MVP. A system built only for Visual House today may need costly restructuring to support multiple providers tomorrow. We raise this now so the architecture decision is made consciously — not discovered painfully after launch.

05 — Proposed Architecture

What We Would
Build and Why

Layer 01

The Clinical Core

Backend Foundation
  • Patient registration with diagnosis and assigned professional
  • Therapist builds personalized therapy program — selecting exercises, setting parameters, defining the weekly schedule
  • System records every session — completed, missed, time spent, date
  • Adherence and progress data flows back to the therapist dashboard for clinical review
  • Reporting engine — filtered by patient, therapist, or date range — exported as PDF or Excel
  • Role-based authentication across three user types: Patient, Orthoptist, Administrator
  • Encrypted medical records with basic change history audit trail
  • Daily automated backups
  • Ley 1581 de 2012 compliance architected from day one — not added later
Stack: Node.js / PostgreSQL / Cloud (AWS or equivalent — pending evaluation together)
Layer 02

The Exercise Engine

The Heart of The Product
  • Four exercise types built for MVP: Smooth Tracking, Saccades, Basic Vergences, Basic Accommodation
  • Each exercise is a precise visual stimulus rendered on screen — not a video, not a pre-recorded animation, but a real-time interactive environment the patient engages with
  • Every exercise is configurable by the therapist: speed, size, duration, difficulty level
  • Session controls — start, pause, resume, complete — so the patient or a supervising parent can manage the experience
  • Built for Chrome on PC — deliberately matched to the clinical constraint to ensure consistent, predictable rendering
  • Smooth, lag-free performance — because a stutter in a visual therapy exercise is not a UI issue, it is a clinical issue
Stack: HTML5 Canvas + requestAnimationFrame + React session wrapper
Layer 03

The Dual Interface

Two UX Philosophies, One System
  • Patient experience: Log in, see today's session, run the exercises, complete. Simple enough for a child, clear enough for a parent helping nearby. No decisions required — just follow the program.
  • Therapist experience: See your patients. See their diagnosis. See what program they are on, what the schedule is, what the adherence looks like. The data tells the story — the therapist makes the clinical decision. Assign programs, adjust parameters, pause or progress treatment.
  • Administrator experience: Manage users, manage system settings, pull global reports across all patients and therapists.
  • Language: Spanish — with English-ready architecture for future expansion
  • Platform: Web, Chrome, PC — built to the clinical constraint, not by default
Stack: React / Tailwind / Recharts

MVP Delivers

  • ✓ Functional exercise engine — four clinically configurable exercise types
  • ✓ Complete patient management — registration, diagnosis, assigned professional
  • ✓ Therapy program builder — exercises, parameters, weekly schedule
  • ✓ Session logging — completed, missed, time spent
  • ✓ Adherence and progress reporting — filtered, exportable as PDF and Excel
  • ✓ Three user roles fully operational — Patient, Orthoptist, Administrator
  • ✓ Home therapy kit flow — in-person patients purchase the kit and access the platform from home
  • ✓ Security and data compliance from day one
  • ✓ 500–1,000 user capacity, 24/7 availability

MVP Does Not Include (By Design)

  • ○ Gamification — Phase 2
  • ○ Adaptive difficulty algorithms — Phase 2. In the MVP, the therapist controls all parameter adjustments manually
  • ○ AI adjustment engine — Phase 2
  • ○ WhatsApp / Calendar integration — Phase 2
  • ○ Mobile / multi-browser support — Phase 2
  • ○ Automated diagnosis — never. Clinical responsibility stays with the orthoptist
  • ○ Virtual Reality — out of scope

How Every Role
Interacts With The System

A step-by-step walkthrough of how each user — patient, therapist, and administrator — experiences the platform from first login to daily use. Each action defined, each behavior specified.

The Patient
Account created by the therapist after in-person assessment — the patient never self-registers. They log in and see one thing: today's session. How many exercises, what to do, start. The Administrator
Global visibility — total patients, active programs, overall adherence, active therapists. Manages all user accounts. Pulls cross-system reports that span every therapist and every patient. Views the audit trail — a chronological record of who did what and when, for accountability and data protection compliance.
